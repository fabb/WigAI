# Story 4.3: Observer Wiring and MCP Server Restart

**Epic:** Epic 4 – Extension UI Settings Integration  
**Story ID:** 4.3  
**Priority:** High  
**Estimated Effort:** 1 day  
**Status:** Planned

## User Story

As a WigAI operator, I want the MCP server to restart automatically whenever host or port settings change so configuration updates take effect immediately.

## Detailed Requirements

1. Add observer registration and notification to `PreferencesBackedConfigManager`, firing events whenever host or port preferences are mutated.
2. Update `McpServerManager` to implement `ConfigChangeObserver` and restart the embedded MCP server with the new host/port.
3. Debounce restarts so sequential host+port edits triggered by the same UI interaction only restart once.
4. Log restart attempts, successes, and failures, including the updated connection URL.
5. If a restart fails, revert to the previous working configuration and notify via log/error.

## Technical Notes

- Ensure observer callbacks run on the Bitwig controller thread; if asynchronous work is required, queue it safely.
- When restarting the MCP server, close the existing server cleanly before opening a new one to avoid port binding issues.
- Consider using a small scheduled task or guard flag to prevent double restarts when both host and port change simultaneously.

## Acceptance Criteria

1. Changing host or port in the preferences UI triggers exactly one MCP server restart, visible in logs.
2. Observers can be added/removed without causing duplicate notifications or memory leaks.
3. Failed restarts are logged with the reason, and the config rolls back to the last known good values.
4. Merely opening the preferences or reading values does not trigger a restart.

## Definition of Done

- [ ] Automated tests cover observer registration, notification, and restart triggering.
- [ ] Manual verification shows the server restarts immediately after editing host/port.
- [ ] Logs include the connection URL for each restart.
- [ ] Code review completed.

## Dependencies

- Story 4.2 (Preferences-backed configuration) must be merged so observers can use its implementation.
