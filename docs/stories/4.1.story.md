# Story 4.1: Extract Config Interfaces

**Epic:** Epic 4 – Extension UI Settings Integration  
**Story ID:** 4.1  
**Priority:** High  
**Estimated Effort:** 2-3 hours  
**Status:** Planned

## User Story

As a WigAI developer, I need a clean `ConfigManager` abstraction so that new configuration mechanisms (like Bitwig Preferences) can be introduced without destabilizing the current runtime.

## Detailed Requirements

1. Extract a `ConfigManager` interface that exposes `getMcpHost()`, `getMcpPort()`, `setMcpHost(String)`, `setMcpPort(int)`, and `addObserver(ConfigChangeObserver)`.
2. Introduce a `ConfigChangeObserver` interface with callbacks `onHostChanged(String oldHost, String newHost)` and `onPortChanged(int oldPort, int newPort)`.
3. Update all current consumers (`McpServerManager`, `WigAIExtension`, tests) to depend on the interface while continuing to use the legacy implementation.
4. Keep the legacy configuration logic (e.g., `LegacyConfigManager`) intact until future stories replace it, guaranteeing zero behavioral change in this story.

## Technical Notes

- Place interfaces under `io.github.fabb.wigai.config`.
- Provide a simple observer registry on the legacy implementation even if it does not yet emit events (future stories will).
- Confirm Gradle/IDE builds succeed with the refactor before proceeding.

## Acceptance Criteria

1. `ConfigManager` and `ConfigChangeObserver` interfaces exist in the codebase with the specified methods.
2. All production classes compile against the interface, and existing functionality (transport commands, MCP server, etc.) works unchanged.
3. Basic unit or smoke tests show configuration values can still be read/written via the legacy implementation.
4. No Bitwig Preferences API usage or server restart logic is introduced in this story.

## Definition of Done

- [ ] Interfaces checked in with JavaDoc or brief comments describing their intent.
- [ ] Legacy configuration class implements the new interfaces.
- [ ] Tests (existing or new) pass locally.
- [ ] Code review completed and approved.

## Dependencies

- None; this is the foundational refactor other configuration stories depend on.
